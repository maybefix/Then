# 要件整理と不足リスト

作成日: 2026-06-02

## 目的

Markdownスニペットエディタをプロトタイプから実用アプリへ進めるために、現時点で決まっている要件、不足している仕様、実装が必要な項目を整理する。

このドキュメントは確定仕様ではなく、今後ユーザー補足を受けて更新していくためのたたき台とする。

## 棚卸方針

現在は実装済み項目と未実装項目が混ざっているため、今後は以下の優先度で管理する。

- P0: Markdownエディタとして成立するために必須。未実装だと実用できない。
- P1: 日常利用で破綻しないために必要。P0完了後に順次実装する。
- P2: 実用性、快適性、保守性を上げる項目。P0/P1の基盤が固まってから実装する。

現時点の最重要方針:

1. 本文は `.md` ファイルとして保存・読み込みできるようにする。
2. パンくずリストを実データ構造に接続し、フォルダ内のMarkdownファイル管理に使えるようにする。
3. スニペットは本文と別管理にするが、本文への挿入後はプレーンテキストとして扱い、元スニペットとの関連は保持しない。

## 現時点で追加された確定要件

- Tauriデスクトップソフトとして作る
- エディタのフォント変更機能を持たせる
- タイプライタースクロールを実装する

## 仕様として詰める必要がある項目

### 1. アプリの最小完成形

- このアプリは「汎用Markdownエディタ」なのか、「小説・本文執筆向けエディタ」なのか
- スニペットは補助素材なのか、本文構成要素なのか
- 最初の完成ラインをどこに置くか

補足待ち:

- 最初に完成させたい利用シーン
- 想定する主ユーザー
- MVPに必須の操作

### 2. Tauriデスクトップソフトとしての範囲

- Webアプリではなく、Tauri製のローカルデスクトップアプリとして提供する
- OSファイルダイアログ、ローカル保存、アプリ設定などをTauri/Rust側で扱う必要がある
- 現時点ではWindowsのみ
- インストーラを作成する
- アプリ設定をどこまで持つかは、現時点ではフォントサイズ・フォント設定・タイプライタースクロールのオフセット値くらい？

### 3. ドキュメント単位

- 1ファイル = 1 Markdown とする
- フォルダ単位で構造化する
- パンくずリスト上でフォルダを右クリックするとノート作成メニューが可能。ノート名を右クリックすると削除メニューが可能。
- パンくずの「第一章 準備 > 剣士の目覚め」（例）は実データ構造とする
- ファイル名を画面上のタイトルとする

### 4. 保存方式

- `.md` ファイルへ保存する
- SQLiteなどDBへ保存するのか、ファイルとDBを併用するのかは要検討
- 自動保存を行う
- 本文の保存先は指定したフォルダ（指定がなければAppのフォルダ）
- スニペットの保存先も指定したフォルダ（指定がなければAppのフォルダ）
- バックアップや履歴は残せたら残したい

### 5. スニペットの扱い

- スニペットはフォルダ内に保存する
- 本文ファイルとは別立てのmarkdownファイルが望ましいが、難しければjsonで
- タイトル、本文、タグ、カテゴリ、文字数以外に必要な属性はとくになし
- 挿入後は元スニペットとの関連を保持しない
- ただの文章として本文に入ればよい
- スニペットの再編集UIが必要

### 6. 挿入されたスニペットの見た目

- 左罫線などの装飾をMarkdownとして永続化しない
- スニペットは記法をつけない（プレーンテキストとしての挿入）

### 7. Markdown編集体験

- WYSIWYG中心でいく
- Markdown記法を直接見せる場面が必要
- 見出し、太字、箇条書き、引用、リンク、コードなどをどこまで初期対応するか
- ショートカットをユーザーで割り振れるとよい
- ツールバーは不要

### 8. フォント変更

- エディタ本文のフォントを変更できるようにする
- フォントサイズ、行間、文字間、段落間隔も設定対象に含めるか検討する
- 変更対象は本文、UI両方とも選択できるようにする（それぞれのフォントを選べるようにする）
- フォント候補はOSフォント一覧から選ぶ
- フォントサイズ、行間も設定対象にする

### 9. タイプライタースクロール

- 入力中のカーソル位置を画面内の一定位置に保つスクロール挙動を実装する
- Milkdown/ProseMirrorの選択更新とスクロール制御を連携する必要がある
- 通常スクロールとの切り替え、固定位置、余白量を決める必要がある
- 設定でオン/オフする
- カーソルを画面中央に保つか、やや上に保つかは設定を可能とする
- 手動スクロール時の挙動

### 10. Markdownプレビュー

- MilkdownのWYSIWYGだけで十分
- プレビューはリアルタイム

### 11. ファイルツリー / プロジェクト管理

- 左サイドバーは不要
- フォルダ、章、本文一覧はパンくずリストから
- 新規作成、リネーム、削除、並び替えが必要

### 12. 履歴

- 右上の履歴ボタンは何を意味するか
- バージョン履歴である

### 13. 検索

- 現状はスニペット検索のみ
- 本文検索が必要
- 全文検索、置換、タグ検索が必要
- プロジェクト内の横断検索もできるようにしたい

## 実装として不足している項目

### P0-1. 本文の `.md` ファイル保存・読み込み

- 実装済み。Tauri側でOSファイルダイアログ、ファイル読み込み、ファイル書き込みを扱う
- 実装済み。単一 `.md` ファイルの開く/保存/名前を付けて保存ができる
- 実装済み。フロント側で「現在開いているファイルパス」「保存済み/未保存」「新規ファイル」を状態として持つ

### P0-2. パンくずリストによるフォルダ内ファイル管理

- 実装済み。パンくずは固定表示ではなく実データ構造に接続している
- 実装済み。フォルダを開き、その配下のサブフォルダ/Markdownファイルをパンくずから辿れる
- 実装済み。パンくずメニューから新規ファイル、新規フォルダを作成できる
- 実装済み。ファイル/フォルダのリネーム、削除ができる
- 実装済み。パンくずメニュー内の上下ボタンで同一フォルダ内の子要素を並び替えできる
- 実装済み。並び順はプロジェクトルートの `.brew/project.json` に保存する
- 残課題: ドラッグによる並び替え、リネーム時の order 名更新、不要 order エントリ掃除

### P0-3. 保存状態とエラー状態

- 実装済み。読み込み中、保存中、未保存、保存失敗をユーザーに明示する

### P1-1. スニペット保存方式の分離

- 本文ファイルとは別立てのMarkdownファイルが望ましいが、難しければJSONでよい

### P1-2. スニペット再編集UIと挿入体験

## 実装済みとして扱う項目

### `.md` ファイル読み書き

- Tauri command により `.md` ファイルを開く、保存、名前を付けて保存できる
- 保存状態表示と自動保存は現在開いているファイルと連動する

### 階層パンくずとプロジェクトファイル管理

- フォルダ配下のサブフォルダと `.md` ファイルを再帰的なツリーとして扱う
- パンくずから子フォルダ/Markdownファイルを辿れる
- 新規ファイル作成、新規フォルダ作成、リネーム、削除を実装済み
- 同一フォルダ内の上下並び替えを実装済み
- 並び順はプロジェクトルートの `.brew/project.json` に保存する

### MilkdownからMarkdown文字列を取り出す処理

- `listenerCtx.markdownUpdated` によりMarkdown文字列を取得している
- 保存対象を `.md` ファイルへ切り替える際も、このMarkdown文字列を使う

### スニペットのCRUD

- 作成、編集、削除、上下並び替え、検索は実装済み
- 現状の永続化先はアプリ状態JSON

### フォント変更

- 本文フォント、UIフォント、文字サイズ、行間を設定可能
- Tauri実行時はWindowsのOSフォント一覧を取得している

### タイプライタースクロール

- オン/オフと固定位置設定は実装済み
- 日本語入力、長文、ドラッグ&ドロップとの干渉確認は未完了
